F.1. adminpack - pgAdmin 지원 툴팩 #

adminpack은 서버 로그 파일의 원격 관리와 같은 추가 기능을 제공하기 위해 pgAdmin 및 기타 관리 및 관리 도구가 사용할 수 있는 여러 지원 기능을 제공합니다. 이러한 모든 기능의 사용은 기본적으로 데이터베이스 수퍼유저에게만 허용되지만 GRANT 명령을 사용하여 다른 사용자에게 허용할 수 있습니다.

표 F.1에 표시된 함수는 서버를 호스팅하는 컴퓨터의 파일에 대한 쓰기 권한을 제공합니다. (읽기 전용 액세스를 제공하는 표 9.101의 함수도 참조) 사용자가 수퍼유저이거나 해당 함수에 적합한 pg_read_server_files 또는 pg_write_server_files 역할 중 하나의 권한이 부여되지 않은 경우 데이터베이스 클러스터 디렉터리 내의 파일에만 액세스할 수 있지만 상대 또는 절대 경로가 허용될 수 있습니다.

표 F.1. 관리자 팩 함수

함수

설명

pg_catalog.pg_file_write ( 파일명 텍스트, 데이터 텍스트, append 부울 ) → bigint

텍스트 파일을 쓰거나 추가합니다.

pg_catalog.pg_file_sync ( 파일명 텍스트 ) → void

파일 또는 디렉터리를 디스크에 플러시합니다.

pg_catalog.pg_file_rename ( 이전 이름 텍스트, 새 이름 텍스트 [, 보관 이름 텍스트 ] ) → boolean

파일 이름을 바꿉니다.

pg_catalog.pg_file_unlink ( 파일명 텍스트 ) → boolean

파일을 제거합니다.

pg_catalog.pg_logdir_ls () → setof record

로그 디렉터리에 있는 로그 파일을 나열합니다.


pg_file_write는 파일명으로 지정된 파일에 지정된 데이터를 씁니다. append가 거짓이면 파일이 이미 존재하지 않아야 합니다. append가 참이면 파일이 이미 존재할 수 있으며, 존재할 경우 추가됩니다. 쓰여진 바이트 수를 반환합니다.

pg_file_sync 파일 이름으로 지정된 파일 또는 디렉터리를 동기화합니다. 실패하면 오류가 발생합니다(예: 지정한 파일이 없는 경우). data_sync_retry는 이 함수에 영향을 미치지 않으므로 데이터베이스 파일을 플러시하지 못하더라도 PANIC 수준의 오류가 발생하지 않습니다.

pg_file_rename은 파일 이름을 바꿉니다. archivename이생략되거나 NULL이면 이전이름을 새 이름으로 바꾸고(이미 존재하지 않아야 함), archivename이 제공되면 먼저 새 이름을 이전 이름으로 바꾸고(이미 존재하지 않아야 함), 그 다음 이전이름을 새 이름으로 바꿉니다. 두 번째 이름 바꾸기 단계에 실패하면 오류를 보고하기 전에 archivename의 이름을 newname으로 다시 바꾸려고 시도합니다. 성공하면 true를 반환하고, 소스 파일이 없거나 쓰기 불가능한 경우 false를 반환하며 그 외의 경우에는 오류를 발생시킵니다.

pg_file_unlink는 지정된 파일을 제거합니다. 성공하면 참을 반환하고, 지정된 파일이 없거나 unlink() 호출에 실패하면 거짓을 반환하며, 그 외의 경우에는 오류를 발생시킵니다.

pg_logdir_ls는 log_directory디렉터리에 있는 모든 로그 파일의 시작 타임스탬프와 경로 이름을 반환합니다. 이 함수를 사용하려면 로그_파일명 매개변수가 기본 설정(postgresql-%Y-%m-%d_%H%M%S.log)이어야 합니다.
