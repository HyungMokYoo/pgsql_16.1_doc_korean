설명

GRANT 명령에는 데이터베이스 개체(테이블, 열, 뷰, 외부 테이블, 시퀀스, 데이터베이스, 외부 데이터 래퍼, 외부 서버, 함수, 프로시저, 절차 언어, 큰 개체, 구성 매개 변수, 스키마, 테이블 스페이스 또는 유형)에 대한 권한을 부여하는 것과 역할의 멤버십을 부여하는 두 가지 기본 변형이 있습니다. 이러한 변형은 여러 가지 면에서 유사하지만 별도로 설명할 만큼 충분히 다릅니다.

데이터베이스 개체의 GRANT

이 GRANT 명령의 변형은 하나 이상의 역할에 데이터베이스 개체에 대한 특정 권한을 부여합니다. 이러한 권한은 이미 부여된 권한에 추가됩니다(있는 경우).

키워드 PUBLIC은 나중에 생성될 수 있는 역할을 포함하여 모든 역할에 권한을 부여할 것을 나타냅니다. PUBLIC은 항상 모든 역할을 포함하는 암묵적으로 정의된 그룹으로 생각할 수 있습니다. 특정 역할은 해당 역할에 직접 부여된 권한, 현재 소속된 모든 역할에 부여된 권한, PUBLIC에 부여된 권한을 합한 권한을 갖게 됩니다.

부여 옵션이 지정되어 있으면 권한을 받은 사람이 다른 사람에게 권한을 부여할 수 있습니다. 부여 옵션이 없으면 수신자는 그렇게 할 수 없습니다. 공개에는 부여 옵션을 부여할 수 없습니다.

GRANTED BY가 지정된 경우 지정된 부여자는 현재 사용자여야 합니다. 이 절은 현재 SQL 호환성을 위해서만 이 형식에 존재합니다.

기본적으로 소유자에게 모든 권한이 있으므로 개체의 소유자(일반적으로 개체를 만든 사용자)에게 권한을 부여할 필요가 없습니다. (그러나 소유자는 안전을 위해 자신의 권한 중 일부를 취소하도록 선택할 수 있습니다.

개체를 삭제하거나 어떤 식으로든 정의를 변경할 수 있는 권한은 부여 가능한 권한으로 취급되지 않으며, 소유자에게 고유한 권한이므로 부여하거나 취소할 수 없습니다. 그러나 개체를 소유한 역할의 멤버십을 부여하거나 취소하면 비슷한 효과를 얻을 수 있습니다(아래 참조). 소유자는 개체에 대한 모든 권한 부여 옵션도 암시적으로 가지고 있습니다.

모든 권한

객체 유형에 사용할 수 있는 모든 권한을 부여합니다. PRIVILEGES 키워드는 PostgreSQL에서는 선택 사항이지만 엄격한 SQL에서는 필수입니다.

일반 함수, 집계 함수 및 창 함수에는 FUNCTION 구문이 작동하지만 프로시저에는 작동하지 않으므로 PROCEDURE를 사용합니다. 또는 정확한 유형에 관계없이 함수, 집계 함수, 창 함수 또는 프로시저를 참조하려면 ROUTINE을 사용합니다.

하나 이상의 스키마 내에서 동일한 유형의 모든 개체에 대한 권한을 부여하는 옵션도 있습니다. 이 기능은 현재 테이블, 시퀀스, 함수 및 프로시저에 대해서만 지원됩니다. ALL TABLES는 특정 개체 GRANT 명령과 마찬가지로 뷰 및 외부 테이블에도 영향을 줍니다. ALL FUNCTIONS는 특정 개체 GRANT명령과 마찬가지로 집계 및 창 함수에도 영향을 주지만 프로시저에는 영향을 주지 않습니다. 프로시저를 포함하려면 ALL ROUTINES를 사용합니다.

역할에 대한 GRANT

GRANT 명령의 이 변형은 하나 이상의 다른 역할에 역할의 멤버십을 부여하고 멤버십 옵션 SET, INHERIT 및 ADMIN을 수정합니다(자세한 내용은 22.3절을참조하세요). 역할의 멤버십은 각 구성원에게 역할에 부여된 권한에 액세스할 수 있고 잠재적으로 역할 자체를 변경할 수 있는 권한을 부여할 수 있다는 점에서 중요합니다. 그러나 실제 부여되는 권한은 부여와 관련된 옵션에 따라 달라집니다. 기존 멤버십의 옵션을 수정하려면 업데이트된 옵션 값으로 멤버십을 지정하기만 하면 됩니다.

아래에 설명된 각 옵션은 TRUE 또는 FALSE로 설정할 수 있습니다. OPTION 키워드는 TRUE의 동의어로 허용되므로 WITH ADMIN OPTION은 WITH ADMIN TRUE의 동의어입니다. 기존 멤버십을 변경할 때 옵션을 생략하면 현재 값이 유지됩니다.

ADMIN 옵션을 사용하면 회원은 다른 사람에게 해당 역할의 멤버십을 부여하고 해당 역할의 멤버십을 취소할 수 있습니다. 관리자 옵션이 없으면 일반 사용자는 이 작업을 수행할 수 없습니다. 역할 자체에 관리자 옵션 이 있는 것으로 간주되지 않습니다. 데이터베이스 수퍼유저는 모든 역할의 멤버십을 누구에게나 부여하거나 취소할 수 있습니다. 이 옵션의 기본값은 FALSE입니다.

상속 옵션은 새 멤버십의 상속 상태를 제어합니다(상속 관련 자세한 내용은 22.3절을 참조하세요). TRUE로 설정하면 새 멤버가 부여된 역할을 상속받게 됩니다. FALSE로 설정하면 새 멤버가 상속받지 않습니다. 새 역할 멤버십을 만들 때 지정하지 않으면 추가되는 역할의 상속 속성이 기본값으로 적용됩니다.

SET 옵션이 TRUE로 설정된 경우 멤버는 SET ROLE명령을 사용하여 부여된 역할로 변경할 수 있습니다. 역할이 다른 역할의 간접 구성원인 경우 각 역할에 SET TRUE가 있는 부여 체인이 있는 경우에만 SET ROLE을 사용하여 해당 역할로 변경할 수 있습니다. 이 옵션의 기본값은 TRUE입니다.

다른 역할이 소유하는 개체를 만들거나 기존 개체의 소유권을 다른 역할에 부여하려면 해당 역할에 SET ROLE을 사용할 수 있어야 하며, 그렇지 않은 경우 ALTER ... OWNER TO 또는 CREATE DATABASE ... OWNER와같은 명령은 실패합니다. 그러나 역할의 권한을 상속받았지만 해당 역할에 대한 SET ROLE 기능이 없는 사용자는 해당 역할이 소유한 기존 개체를 조작하여 역할에 대한 전체 액세스 권한을 얻을 수 있습니다(예: 트로이 목마처럼 작동하도록 기존 함수를 재정의할 수 있음). 따라서 역할의 권한은 상속되지만 SET ROLE을 통해 액세스할 수 없어야 하는 경우 해당 역할은 SQL 개체를 소유하지 않아야 합니다.

GRANTED BY를 지정하면 부여가 지정된 역할에 의해 수행된 것으로 기록됩니다. 사용자는 해당 역할의 권한을 보유한 경우에만 다른 역할에 권한을 부여할 수 있습니다. 부여자로 기록된 역할은 부트스트랩 수퍼유저가 아닌 한 대상 역할에 대해 ADMIN OPTION이 있어야 합니다. 부트스트랩 수퍼유저가 아닌 다른 부여자가 있는 것으로 기록된 권한은 부여자가 해당 역할에 대한 ADMIN OPTION을 계속 보유하고 있는지에 따라 달라지므로 ADMIN OPTION이 취소되면 종속된 부여도 취소되어야 합니다.

권한의 경우와 달리 역할의 멤버십은 PUBLIC에게 부여할 수 없습니다. 또한 이 형식의 명령에서는 role_specification에 노이즈 단어 GROUP을허용하지 않습니다.

참고

REVOKE 명령은 액세스 권한을 취소하는 데 사용됩니다.

PostgreSQL 8.1부터 사용자와 그룹의 개념이 역할이라는 단일 엔티티로 통합되었습니다. 따라서 더 이상 GROUP키워드를 사용하여 수권자가 사용자 또는 그룹인지 식별할 필요가 없습니다. GROUP은여전히 명령에 허용되지만 노이즈 단어입니다.

사용자가 특정 열 또는 전체 테이블에 대해 해당 권한을 보유한 경우 열에 대해 SELECT, INSERT 등을 수행할 수 있습니다. 테이블 수준에서 권한을 부여한 다음 한 열에 대해 권한을 취소해도 테이블 수준 부여는 열 수준 작업의 영향을 받지 않습니다.

개체의 소유자가 아닌 사용자가 개체에 대한 권한을 부여하려고 할 때 사용자에게 개체에 대한 권한이 전혀 없는 경우 명령이 완전히 실패합니다. 일부 권한을 사용할 수 있는 한 명령은 계속 진행되지만 사용자에게 부여 옵션이 있는 권한만 부여합니다. 부여 옵션이 없는 경우 GRANT ALL PRIVILEGES 양식은 경고 메시지를 표시하고, 다른 양식은 명령에 구체적으로 지정된 권한에 대한 부여 옵션이 없는 경우 경고를 표시합니다. (원칙적으로 이러한 문구는 개체 소유자에게도 적용되지만 소유자는 항상 모든 부여 옵션을 보유한 것으로 취급되므로 이러한 경우가 발생할 수 없습니다.)

데이터베이스 수퍼유저는 개체 권한 설정에 관계없이 모든 개체에 액세스할 수 있다는 점에 유의해야 합니다. 이는 유닉스 시스템에서 루트의 권한과 비슷합니다. 루트와 마찬가지로 꼭 필요한 경우를 제외하고는 수퍼유저로 활동하는 것은 현명하지 않습니다.

수퍼유저가 부여 또는 취소명령을 실행하면 해당 명령은 영향을 받는 개체의 소유자가 실행하는 것처럼 수행됩니다. 특히 이러한 명령을 통해 부여된 권한은 개체 소유자가 부여한 것으로 나타납니다. (역할 멤버십의 경우 부트스트랩 수퍼유저가 멤버십을 부여한 것으로 나타납니다.)

영향을 받는 개체의 소유자는 아니지만 개체를 소유하는 역할의 구성원이거나 개체에 대해 부여 옵션이 있는 권한을 가진 역할의 구성원인 역할도 GRANT 및 REVOKE를 수행할 수 있습니다. 이 경우 권한은 실제로 개체를 소유하거나 부여 옵션이 있는 권한을 가진 역할이 부여한 것으로 기록됩니다. 예를 들어 테이블 t1을 역할 g1이 소유하고 있고 역할 u1이 구성원인 경우, u1은 u2에게 t1에 대한 권한을 부여할 수 있지만 해당 권한은 g1이 직접 부여한 것으로 나타납니다. 역할 g1의 다른 구성원은 나중에 해당 권한을 취소할 수 있습니다.

GRANT를 실행하는 역할이 둘 이상의 역할 멤버십 경로를 통해 필요한 권한을 간접적으로 보유하고 있는 경우 어느 역할이 포함된 역할이 권한을 부여한 것으로 기록될지는 지정되지 않습니다. 이러한 경우 GRANT를 수행할 특정 역할로 SET ROLE을 사용하는 것이 가장 좋습니다.

테이블에 대한 권한을 부여해도 SERIAL 열에 연결된 시퀀스를 포함하여 테이블에서 사용하는 모든 시퀀스에 대한 권한이 자동으로 확장되지는 않습니다. 시퀀스에 대한 권한은 별도로 설정해야 합니다.

특정 권한 유형에 대한 자세한 내용과 개체의 권한을 검사하는 방법은 5.7 섹션을 참조하세요.

호환성

SQL 표준에 따르면 모든 권한에 PRIVILEGES키워드는 필수입니다. SQL 표준은 명령당 하나 이상의 개체에 대한 권한 설정을 지원하지 않습니다.

예를 들어, 테이블 소유자는 자신의 INSERT, UPDATE, DELETE, TRUNCATE권한을 취소하여 테이블을 자신에게만 읽기 전용으로 설정할 수 있습니다. SQL 표준에 따르면 이는 불가능합니다. 그 이유는 소유자의 권한을 소유자가 자신에게 부여한 것으로 간주하기 때문에 소유자도 권한을 취소할 수 있기 때문입니다. SQL 표준에서 소유자의 권한은 가정된 엔티티인 "_SYSTEM"에 의해 부여됩니다. " _SYSTEM"이 아닌 소유자는 이러한 권한을 취소할 수 없습니다.

SQL 표준에 따르면 부여 옵션은 PUBLIC에 부여할 수 있지만 PostgreSQL은 역할에 부여 옵션만 지원합니다.

SQL 표준에 따르면 GRANTED BY 옵션은 CURRENT_USER 또는 CURRENT_ROLE만 지정할 수 있습니다. 다른 변형은 PostgreSQL 확장입니다.

SQL 표준은 문자 집합, 콜레이션, 번역 등 다른 종류의 개체에 대한 USAGE 권한을 제공합니다.

SQL 표준에서 시퀀스에는 USAGE권한만 있으며, 이 권한은 PostgreSQL의 nextval 함수에 해당하는 NEXT VALUE FOR식의 사용을 제어합니다. 시퀀스 권한 SELECT 및 UPDATE는 PostgreSQL 확장입니다. currval 함수에 대한 시퀀스 USAGE 권한의 적용도 함수 자체와 마찬가지로 PostgreSQL 확장입니다.

데이터베이스, 테이블스페이스, 스키마, 언어 및 구성 매개변수에 대한 권한은 PostgreSQL 확장입니다.
